# Agent Guide

This file gives AI coding agents enough context to work on this project without rediscovering the deployment model from scratch.

## Project Summary

This is a Vue 3 + Vite static app meant to be built locally, uploaded to OCI Object Storage, and exposed through OCI API Gateway.

The app is intentionally small:

- `src/App.vue` renders the UI and calls `GET /health`.
- `src/main.ts` mounts Vue.
- `src/styles.css` contains all styling.
- `vite.config.ts` sets `base: './'` so built assets work behind an API Gateway deployment path.
- `dist/` is generated by `npm run build` and must not be committed.

## Important Runtime Behavior

The app does not require `VITE_API_BASE_URL`.

By default, the browser computes the current API Gateway deployment URL from `window.location` and calls:

```text
{current-deployment-url}/health
```

`VITE_API_BASE_URL` is only an optional build-time override.

## OCI Deployment Model

The recommended OCI setup follows the public Oracle tutorial for hosting a static website with API Gateway and Object Storage.

Use a private bucket plus two Pre-Authenticated Requests:

1. One PAR for `index.html`.
2. One PAR for the bucket, or preferably the `assets/` prefix, to read static assets.

Agents should pay attention to the PAR expiration date. If the PAR expires, API Gateway will stop being able to fetch the files.

API Gateway should have:

```text
Path: /{req*}
Methods: GET
Backend type: Dynamic routing
```

Suggested dynamic backends:

```text
Default backend:
URL: INDEX_PAR_URL
```

```text
JS backend:
Match type: Wildcard
Match expression: *.js
URL: BUCKET_PAR_BASE_URL/${request.path[req]}
```

```text
CSS backend:
Match type: Wildcard
Match expression: *.css
URL: BUCKET_PAR_BASE_URL/${request.path[req]}
```

If the app gains more asset types, add rules for extensions such as `*.png`, `*.svg`, `*.woff2`, and `*.ico`.

The deployment should also include:

```text
Path: /health
Methods: GET
Backend type: Stock response
Status code: 200
Body: {"status":"ok"}
Header: Content-Type = application/json
```

## Optional OCI CLI Upload

An agent may upload `dist/` with OCI CLI if the user explicitly wants that and the local environment is authenticated.

Do not hardcode real OCIDs, namespaces, bucket names, PAR tokens, or profile names in committed files.

Example variables for a shell session:

```bash
export OCI_PROFILE=SEU_PROFILE
export OCI_NAMESPACE=SEU_NAMESPACE
export OCI_BUCKET=SEU_BUCKET
```

For federated OCI CLI sessions, commands often need:

```text
--auth security_token
```

Upload `index.html`:

```bash
oci os object put --namespace-name "$OCI_NAMESPACE" --bucket-name "$OCI_BUCKET" --file dist/index.html --name index.html --content-type text/html --force --profile "$OCI_PROFILE" --auth security_token
```

Upload CSS files:

```bash
for file in dist/assets/*.css; do oci os object put --namespace-name "$OCI_NAMESPACE" --bucket-name "$OCI_BUCKET" --file "$file" --name "assets/$(basename "$file")" --content-type text/css --force --profile "$OCI_PROFILE" --auth security_token; done
```

Upload JS files:

```bash
for file in dist/assets/*.js; do oci os object put --namespace-name "$OCI_NAMESPACE" --bucket-name "$OCI_BUCKET" --file "$file" --name "assets/$(basename "$file")" --content-type application/javascript --force --profile "$OCI_PROFILE" --auth security_token; done
```

After upload, verify headers:

```bash
oci os object head --namespace-name "$OCI_NAMESPACE" --bucket-name "$OCI_BUCKET" --name index.html --query '{contentType:"content-type",contentLength:"content-length"}' --profile "$OCI_PROFILE" --auth security_token
```

## Safety Notes

- Never commit `.env`, PAR URLs, OCIDs, API keys, session tokens, or tenant-specific values.
- Keep README examples generic.
- Rebuild before uploading after any source change.
- If a browser cannot load assets, check deployment path slash behavior, PAR expiration, API Gateway route matching, and subnet ingress/egress TCP 443.
